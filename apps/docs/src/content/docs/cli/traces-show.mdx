## Synopsis

```bash
apo traces show <trace-id> [--verbose] [--errors-only]
```

## Description

Shows a trace's call details. The argument is a **trace id** (required) or a unique prefix. A trace id is a separate field on a run — `apo runs show` prints it at the bottom of its output.

## Options

| Option | Default | Purpose |
|---|---|---|
| `<trace-id>` | (required) | Trace id or unique prefix. |
| `--verbose` | off | Show per-call input/output and full messages. |
| `--errors-only` | off | Filter to ERROR and WARNING calls only. |

With `--verbose`, each call expands to show its full input and output — the messages sent to the model, the tool results returned, the deliverables collected. With `--errors-only`, only calls that errored or warned are shown, for fast triage.

## Why the trace matters

When a test like `t.calledTool("read_file", { input: { path: "contract.pdf" } })` fails, the trace tells you why. You can see whether the tool was called at all, what arguments it received, and what it returned. The trace is the bridge between a failing assertion and the hypothesis about what to change. See [Traces](/concepts/traces/).

## Example

```bash
# Show a trace by id
apo traces show f1a2b3c4

# Show with full per-call detail
apo traces show f1a2b3c4 --verbose

# Show only what errored
apo traces show f1a2b3c4 --errors-only
```
