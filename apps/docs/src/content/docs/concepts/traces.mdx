**When a run fails, the trace is where you go.** It's the full runtime record of one task run: the call tree, the tokens, the actual messages. It's treated as a first-class surface, not an afterthought.

## What's in a trace

- **The call tree**: every call the agent made, parent/child relationships, what it returned.
- **Token usage**: input/output tokens per call, with cost.
- **Model parameters**: temperature, model, all params actually used.
- **The messages**: the actual system, user, assistant, and tool messages exchanged.

A trace is a tree built over time, not a flat log. The calls nest, and the order matters:

<TraceTreeDemo />

## One trace per run

You never have to figure out which trace goes with a run — each task run owns at most one trace, and there are never two competing traces for one execution. See [Mental model](/concepts/mental-model/) for the rule that enforces this.

Start at the failing test (from `apo runs show`), then walk the call tree to see what the agent actually did. The messages are where "why did it do that" gets answered: you can read exactly what the agent was told and what it said back.
