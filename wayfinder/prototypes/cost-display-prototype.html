Cost display — per-dimension breakdown

Prototype for ticket 11. Every call can now carry 1–6 priceable dimensions (ticket 01). This shows the proposed layouts for the tooltip, the trace-detail costs tab, and the run/session rollup — across every state the cost system produces. Hover any pill to see its tooltip; all tooltips are also shown pinned below for side-by-side comparison.

Tokens match globals.css exactly. Tooltip surface is inverted (bg-foreground text-background) like tooltip.tsx. Dimensions use monochrome shades, not hues — color is state per design.md, and a dimension is identity, not state.

Decision candidates for the tooltip: (1) group by family + sort by magnitude — scales 1→6 dims without a layout jump; (2) hide zero-cost rows (a dimension priced at 0 for this call isn't interesting); (3) total above the rule so the eye lands on it first; (4) provenance as a quiet footer line (computed / provided), not a parallel "Provided: $X · Calculated: $Y" row — ticket 06 collapsed that split, the tooltip shouldn't re-show it; (5) unpriced dimensions surfaced in --warning amber so a mispriced model is visible, not silent.

The mix bar is the new affordance. It's 3px tall (present, not loud), monochrome, and uses the same shade-per-dimension key as the tooltip so the two surfaces share one legend. A model that's 90% input (cache miss) vs one that's 60% cache_read (cache hit) look visibly different at a glance — the debuggability the per-dimension breakdown exists to provide. Hovering a segment in production would isolate that dimension; in this throwaway it's static.

3 · Runs & sessions list — compact

List views stay scalar (a single total per run) — a dimension breakdown doesn't belong in a dense table row. The tooltip-on-hover carries the mix for anyone who wants it. Unpriced calls surface with the warning amber instead of being silently zero.

Unpriced = amber, not red. Red (--destructive) is reserved for errors in this design system. A call that genuinely has no price match isn't an error, it's "data we couldn't price" — --warning amber. The ⚠ signals "this total is partial; some calls were unpriced." This is the only new color rule; everything else stays monochrome per design.md.

4 · What this prototype decides

① Group by family, sort by magnitude. Input-family (input, cache_read, cache_write_5m, cache_write_1h) and output-family (output, reasoning) as labeled groups; within each, largest-cost first. Scales 1→6 dims without a layout jump, and reads naturally (biggest cost is most visible).

② Provenance is a quiet footer, not a parallel row. The old "Provided: $X · Calculated: $Y" two-number display is gone — ticket 06 collapsed that into one effective total + a provenance flag. The tooltip honors that: one total, a small computed / provided by SDK line at the bottom. The divergence-flag logic (Math.abs(provided - calculated) > 0.0001) is deleted with it.

③ Client-side recompute is removed. model-pricing.ts's fetchModelPricing / matchModelPricing / computeCallBreakdown are deleted. The dashboard reads stored cost_breakdown (ticket 06) directly. The regex-anchoring mismatch (frontend ^…$ vs backend re.search) dies with it — ticket 10 made matching server-side only.

④ Unpriced dimensions surface in amber. A usage key present in raw_usage but with no price row shows as reasoning · 980 tok — unpriced in --warning, with a ⚠ footer. Makes a mispriced model debuggable after the fact — the stated purpose of storing per-dimension breakdown (locked decision 3).

⑤ Legacy calls render "no breakdown." Pre-migration calls (ticket 09: cost_breakdown/raw_usage null) show the scalar total + a quiet "legacy call · no breakdown stored" footer. No synthesis from current prices (that would be misleading per ticket 04's time-windowing).

⑥ Dimension identity = monochrome shades, never hue. Six grayscale steps for the six canonical keys. design.md says color is state, not decoration; a dimension is identity. Hue is reserved for the one genuine state change: unpriced = amber warning.
